## 2. Core Directives

### 2.1 Architectural Boundary

**`src/app/` is routing only.** Route files (`page.tsx`, `layout.tsx`) must contain zero business logic. They are thin wrappers that compose components imported from feature slices. A page file may:
- Import a component from `src/features/[domain]/components/`
- Pass search params or route params as props
- Call a single top-level query from `src/features/[domain]/queries.ts` for initial data fetching

A page file may NOT:
- Contain `db.` calls directly
- Define Zod schemas
- Contain `useState`/`useEffect` for domain state management (use feature components for that)
- Import from `@prisma/client` or `@/generated/prisma`

**`src/core/ui/` is read-only primitives.** The contents of `src/core/ui/` are generated by `npx shadcn@latest add`. You never manually create, edit, or delete files in this directory. If a required primitive is missing, invoke the shadcn CLI. You may compose higher-order components from these primitives inside `src/features/[domain]/components/`.

**All implementations live in `src/features/[domain]/`.** Every server action, database query, domain component, and validation schema must be placed inside its corresponding feature slice. Cross-feature imports are forbidden—use shared utilities from `src/core/` instead.

### 2.2 State Integrity

**All persistence goes through Prisma.** The single Prisma client instance is exported from `src/core/db/index.ts` as `db`. Every database operation must import and use this client:

```ts
import { db } from "@/core/db";
```

**Never:**
- Instantiate `new PrismaClient()` outside `src/core/db/index.ts`
- Execute raw SQL via `$executeRawUnsafe` or `$queryRawUnsafe` unless absolutely unavoidable (and only with explicit justification in the PR description)
- Bypass Prisma's type system with `as any` or `@ts-ignore` on database payloads
- Store untyped JSON blobs when a typed relation or enum exists in the schema

**Schema is the single source of truth.** The Prisma schema at `prisma/schema.prisma` defines all models, enums, and relations. If a task requires a schema change, modify the schema first, run `npx prisma format`, `npx prisma generate`, and `npx prisma db push` before writing application code.

### 2.3 Git Safety & Multi-Agent Isolation

**Work only within your assigned feature slice.** When executing a task, your file changes must be confined to:
- `src/features/[your-domain]/` (all files within)
- `prisma/schema.prisma` (only if the task explicitly requires a schema change)
- `src/app/[your-route]/` (only the route wrapper for your domain)

**Never modify:**
- Another feature slice's directory
- `src/core/` files (unless the task explicitly requires adding a shared utility or shadcn component)
- Root configuration files (`tsconfig.json`, `next.config.ts`, `tailwind.config.ts`, `components.json`) unless the task explicitly requires it
- `src/app/globals.css` unless adding domain-specific CSS variables approved by the task

**Conflict prevention:** If you need a shared type or utility that does not exist, create it in `src/core/utils/` and document the addition. Do not duplicate utilities inside feature slices.

## 5. Skill: Data Persistence (prisma-db-skill)

### 5.1 Prisma Client Usage

Import the singleton client:
```ts
import { db } from "@/core/db";
```

**Connection architecture:** The client uses `@prisma/adapter-pg` with a `PrismaPg` driver adapter. The connection string is sourced from `DATABASE_URL` in `.env`. The client is cached on `globalThis` in development to prevent connection exhaustion during hot reload.

### 5.2 Query Patterns

**Read operations** go in `queries.ts`:
```ts
"use server";

import { db } from "@/core/db";

export async function getDonorById(donorId: number) {
  return db.donor.findUnique({
    where: { donor_id: donorId },
    include: { collections: true, user: true },
  });
}

export async function getDonorsByStatus(status: "ACTIVE" | "INACTIVE") {
  return db.donor.findMany({
    where: { status },
    orderBy: { registration: "desc" },
  });
}
```

**Write operations** go in `actions.ts`:
```ts
"use server";

import { db } from "@/core/db";
import { revalidatePath } from "next/cache";

export async function createDonor(data: CreateDonorInput) {
  const donor = await db.donor.create({ data });
  revalidatePath("/dashboard/donors");
  return donor;
}
```

### 5.3 Transaction Requirements

Use `$transaction` for any multi-step write that affects critical medical tracking state. The following operations **require** transactional protection:

| Operation | Reason |
|---|---|
| Batch pooling (assigning collections to a batch) | Atomic volume summation must match `Batch.total_volume` |
| Pasteurization status update + lab result recording | `Batch.status` and `LabResult` must be consistent |
| Inventory deduction during dispensing | `Inventory.available_vol` must decrease atomically with `Dispensing` creation |
| Batch disposal | `Batch.status` → `DISPOSED`, `Inventory.available_vol` → 0, and `Disposal` record must be atomic |
| Collection → Batch assignment | `Collection.batch_id` update and batch volume recalculation must be atomic |

**Transaction pattern:**
```ts
export async function dispenseMilk(input: DispenseInput) {
  return db.$transaction(async (tx) => {
    const inventory = await tx.inventory.findUnique({
      where: { batch_id: input.batch_id },
    });

    if (!inventory || inventory.available_vol < input.volume) {
      throw new Error("Insufficient available volume");
    }

    const dispensing = await tx.dispensing.create({
      data: {
        batch_id: input.batch_id,
        beneficiary_id: input.beneficiary_id,
        dispensed_by: input.dispensed_by,
        dispensing_date: new Date(),
        volume: input.volume,
        price: input.price,
        total: input.volume * input.price,
        remarks: input.remarks,
      },
    });

    await tx.inventory.update({
      where: { batch_id: input.batch_id },
      data: { available_vol: { decrement: input.volume } },
    });

    return dispensing;
  });
}
```

## 6. Skill: Input Sanitization (validation-zod-skill)

### 6.1 Zod Validation Mandate

**Every server action and API endpoint must validate input through Zod before executing any database operation.** No exceptions.

Create a `schemas.ts` file inside each feature slice when implementing that slice's actions:

```
src/features/donors/schemas.ts
src/features/collections/schemas.ts
src/features/laboratory/schemas.ts
src/features/inventory/schemas.ts
src/features/dispensing/schemas.ts
src/features/disposal/schemas.ts
src/features/sms/schemas.ts
```

### 6.2 Schema Pattern

```ts
import { z } from "zod";

export const createDonorSchema = z.object({
  first_name: z.string().min(1, "First name is required").max(100),
  last_name: z.string().min(1, "Last name is required").max(100),
  birthdate: z.coerce.date().refine((date) => {
    const age = Math.floor((Date.now() - date.getTime()) / (365.25 * 24 * 60 * 60 * 1000));
    return age >= 18;
  }, "Donor must be at least 18 years old"),
  address: z.string().min(1, "Address is required").max(500),
  contact_no: z.string().regex(/^(\+63|0)[0-9]{10}$/, "Invalid Philippine phone number"),
  status: z.enum(["ACTIVE", "INACTIVE"]).default("ACTIVE"),
});

export type CreateDonorInput = z.infer<typeof createDonorSchema>;
```

### 6.3 Data Dictionary Validation Constraints

| Field | Constraint | Rule |
|---|---|---|
| `Donor.birthdate` | Minimum age 18 years | `z.coerce.date().refine(age >= 18)` |
| `Donor.contact_no` | Philippine phone format | `z.string().regex(/^(\+63\|0)[0-9]{10}$/)` |
| `Collection.ctn` | Collection tracking number | Positive integer, auto-generated |
| `Collection.volume` | Daily milk intake max 1200mL per donor | `z.number().positive().max(1200)` |
| `Collection.program` | Valid program enum | `z.enum(["SUPSUP_TODO", "MILKY_WAY", "MOMS_ACT"])` |
| `Batch.batch_code` | Unique batch identifier | `z.string().min(1).max(50)` |
| `Batch.total_volume` | Must be positive | `z.number().positive()` |
| `LabResult.result` | Valid test result | `z.enum(["PASS", "FAIL", "PENDING"])` |
| `Inventory.available_vol` | Cannot be negative | `z.number().min(0)` |
| `Dispensing.volume` | Must be positive, ≤ available | `z.number().positive()` (further validated in action against inventory) |
| `Dispensing.price` | Must be non-negative | `z.number().min(0)` |
| `Disposal.volume` | Must be positive | `z.number().positive()` |
| `SMS.message` | Max 160 characters (SMS limit) | `z.string().min(1).max(160)` |
| `Beneficiary.contact_no` | Philippine phone format | `z.string().regex(/^(\+63\|0)[0-9]{10}$/)` |
| `User.username` | Alphanumeric, 3-50 chars | `z.string().min(3).max(50).regex(/^[a-zA-Z0-9_]+$/)` |
| `User.password` | Min 8 chars, complexity requirements | `z.string().min(8)` |

### 6.4 Action Integration Pattern

```ts
"use server";

import { db } from "@/core/db";
import { createDonorSchema, type CreateDonorInput } from "./schemas";
import { revalidatePath } from "next/cache";

export async function createDonor(rawInput: unknown) {
  const parsed = createDonorSchema.safeParse(rawInput);

  if (!parsed.success) {
    return { success: false, errors: parsed.error.flatten().fieldErrors };
  }

  const input: CreateDonorInput = parsed.data;

  const donor = await db.donor.create({
    data: {
      user_id: input.user_id,
      first_name: input.first_name,
      last_name: input.last_name,
      birthdate: input.birthdate,
      address: input.address,
      contact_no: input.contact_no,
      status: input.status,
    },
  });

  revalidatePath("/dashboard/donors");
  return { success: true, data: donor };
}
```

## 9. Error Handling Contract

### Server Actions

Server actions must return a consistent result shape:
```ts
type ActionResult<T> =
  | { success: true; data: T }
  | { success: false; errors: Record<string, string[]> };
```

Never throw raw errors from server actions to the client. Catch, shape, and return.

### Database Operations

Wrap critical database operations in try/catch. Log errors to `AuditLog` when appropriate. Never expose Prisma error messages directly to the client — map them to user-friendly messages.

### Transaction Rollback

If a `$transaction` callback throws, Prisma automatically rolls back all operations within that transaction. Do not implement manual rollback logic — let the transaction handle it.

## 10. Security Constraints

- **Password storage:** Passwords must be hashed before storage (use `bcrypt` or `argon2`). Never store plaintext passwords.
- **Session management:** All dashboard routes must verify authentication. Unauthenticated users redirect to `(auth)/login`.
- **Role-based access:** `ADMIN` users have full access. `STAFF` users have domain-specific access. Enforce at the action level, not just the UI level.
- **Audit logging:** Every write operation must create an `AuditLog` entry recording the acting `user_id`, timestamp, and action description.
- **Input sanitization:** Zod schemas handle structural validation. Additional sanitization (XSS prevention, SQL injection) is handled by Prisma's parameterized queries and React's JSX escaping.
- **Environment secrets:** Never commit `.env` contents, API keys, or credentials to version control. The `.gitignore` already excludes `.env*`.

## 12. Git Worktree Workflow for Parallel Feature Execution

### Overview

The human Architect orchestrates parallel feature development using Git worktrees. Each worktree is an isolated checkout on its own branch, enabling multiple agents to work on different FSD domains simultaneously without merge conflicts.

### Worktree Creation

The Architect spawns isolated branches using:

```bash
git worktree add ../agent-workspaces/feature-[domain-name] -b feature/[domain-name]
```

Examples:
```bash
git worktree add ../agent-workspaces/feature-donors -b feature/donors
git worktree add ../agent-workspaces/feature-collections -b feature/collections
git worktree add ../agent-workspaces/feature-inventory -b feature/inventory
```

### Parallel IDE Windows

Each worktree is opened in its own VS Code window. The Architect:

1. Opens the worktree folder as a new VS Code window
2. Assigns a Custom Mode (from `.kilo/modes/`) to the agent in that window
3. Provides the task blueprint based on the FSD domain assignment

### Custom Mode Assignment

| Domain | Recommended Mode | Reason |
|---|---|---|
| `donors` — UI tables, forms | `feature-ui-builder` | Component-heavy domain |
| `donors` — server actions | `prisma-db-engine` | Data access layer |
| `collections` — workflow logic | `prisma-db-engine` | Complex transactional operations |
| `laboratory` — test results | `prisma-db-engine` | Lab result recording |

### Integration Workflow

1. Each agent completes its feature slice independently
2. The Architect reviews each worktree's changes
3. Changes are merged into the main branch one domain at a time
4. Schema changes are merged first (if any), then application code
5. The Architect resolves any cross-domain integration issues

### Conflict Prevention

- Agents work strictly within their assigned `src/features/[domain]/` directory
- Shared types or utilities go in `src/core/utils/` (coordinated by the Architect)
- Schema changes are centralized on the main branch before feature branches diverge
- Never use `git stash` across worktrees — stashes are shared globally

### Cleanup

After merging, remove the worktree:
```bash
git worktree remove ../agent-workspaces/feature-[domain-name]
```
